# Архитектурный стандарт проекта (System-Mechanic-Component-Container)

## 1. Цель документа

Этот документ описывает универсальный подход к построению архитектуры проекта без привязки к конкретной тематике игры.

Базовая единица фичи: `System + Mechanic + Component + Container`.

## 2. Базовые роли

### 2.1 System

`System` управляет жизненным циклом фичи в игровом цикле:

- участвует в общем update-loop;
- проверяет текущее состояние игры (`GameState`);
- решает, должна ли выполняться логика в текущем кадре;
- делегирует непосредственные действия в `Mechanic`.

`System` не должен содержать тяжелую предметную логику и не должен напрямую хранить Unity-ссылки.

### 2.2 Mechanic

`Mechanic` описывает непосредственные действия над сущностями:

- изменение данных;
- выполнение операций фичи;
- запуск переходов состояния (при необходимости);
- реакция на события/команды, пришедшие из `System`.

`Mechanic` не управляет глобальным циклом сам, а вызывается системой.

### 2.3 Component

`Component` хранит данные, связанные с системой:

- runtime-флаги;
- кэшированные значения;
- ссылки на внутренние объекты фичи (не Unity View-ссылки).

`Component` должен быть максимально «данными», без сложной бизнес-логики.

### 2.4 Container

`Container` — мост между архитектурой и Unity:

- хранит ссылки на Unity-компоненты (`Transform`, `Renderer`, UI-компоненты и т.д.);
- предоставляет точку подключения scene-объектов;
- используется для связывания runtime-систем с объектами сцены.

Контейнеры являются инфраструктурой представления и не должны содержать сложную доменную логику.

## 3. Контракты и жизненный цикл

Минимальные контракты:

- `IGameSystem`: `Initialize`, `UpdateSystem`, `DisposeSystem`;
- `IGameMechanic`: `UpdateMechanic`, `DisposeMechanic`.

Поток исполнения:

1. Все системы регистрируются в `GameSystemsHandler`.
2. Вызывается `Initialize` у каждой системы.
3. В общем update-цикле вызывается `UpdateSystem`.
4. Внутри `UpdateSystem` система фильтрует состояние и вызывает `Mechanic.UpdateMechanic`.
5. При остановке вызываются `DisposeSystem`/`DisposeMechanic`.

## 4. Управление состояниями игры

Состояние игры — центральный оркестратор поведения систем.

Правило:

- система должна явно определять, в каких стейтах она активна;
- если стейт не подходит, `System` завершает `UpdateSystem` без выполнения механики.

Пример последовательности стейтов (шаблон):

`Bootstrap -> InitializeSession -> LoadContent -> MainMenu -> PrepareLevel -> Gameplay -> Pause -> EvaluateResult -> Reward -> LevelTransition -> UpdateProgress -> GameOver`
Обязательное правило:

- этот блок не является фиксированным для всех проектов;
- при старте нового проекта последовательность стейтов должна быть адаптирована под его игровой цикл;
- при любом изменении state-flow в ходе разработки этот раздел должен обновляться синхронно с кодом (`GameStates`, переходы систем).

## 5. Как расширять последовательность стейтов

При добавлении нового стейта:

1. Добавить новый элемент в `GameStates`.
2. Определить место в общей последовательности переходов.
3. Явно зафиксировать переходы:
   - `кто` переводит в новый стейт;
   - `при каком условии`;
   - `в какой следующий стейт` происходит выход.
4. Обновить системы, которые должны быть активны в новом стейте.
5. Убедиться, что нет «тупиковых» состояний (без выхода) и циклов без управляющего условия.
6. Обновить архитектурную документацию и таблицу стейтов.
7. При добавлении, удалении или переименовании любого стейта в рабочем проекте обязательно синхронно обновить этот раздел и таблицу переходов.

Рекомендуется поддерживать отдельную таблицу вида:

- `State`;
- `Owner System`;
- `Enter Condition`;
- `Exit Condition`;
- `Next State`.

## 6. Структура папок и добавление новых фич

Рекомендуемая структура внутри `Assets/Scripts`:

- `GameStartupScripts` — bootstrap/инициализация проекта;
- `ScenesOperatingScripts` — управление сценами и связка scene root;
- `GameScripts` — общие сущности и перечисления (`GameStates`);
- `GameSystemsScripts` — все runtime-системы;
- `...Feature...` подпапки — фичи, каждая в собственном модуле;
- `GameObjectFactories` — статические фабрики создания игровых объектов.

Правило для фичи:

- каждая новая фича добавляется отдельным модулем/папкой;
- внутри модуля должны быть минимум:
  - `FeatureSystem.cs`;
  - `FeatureMechanic.cs`;
  - `FeatureComponent.cs`;
  - `FeatureContainer.cs` (если фича работает с Unity-объектами сцены).

## 7. Правила добавления новых фич

1. Определить ответственность фичи и ее границы.
2. Добавить `System/Mechanic/Component` (и `Container` при необходимости).
3. Зарегистрировать систему в bootstrap (`LoadGameComponent` или эквивалент composition root).
4. Подключить контейнер в scene-композиции (`SceneSystemsContainer`/`ISceneRoot` wiring).
5. Привязать систему к нужным стейтам.
6. Добавить `Dispose`-очистку подписок/ресурсов.
7. Обновить архитектурные документы.

## 8. Правила создания игровых объектов

Все новые игровые объекты должны создаваться только через статические фабрики.

Правила:

- запрещено создавать новые игровые объекты напрямую в системах/механиках;
- создание выносится в статические фабрики;
- фабрики хранятся только в `Assets/Scripts/GameObjectFactories`.

Ограничение:

- папка `GameObjectFactories` в проекте должна быть единственной;
- создание других папок с именем `GameObjectFactories` в любых местах проекта запрещено.

## 9. Антипаттерны (что запрещено)

- смешивать логику состояний и Unity-view логику в одном классе;
- хранить Unity-ссылки в `System` и `Mechanic`, если это может быть вынесено в `Container`;
- создавать «монолитную» систему, которая покрывает несколько независимых фич;
- добавлять стейты без явной схемы входа/выхода;
- создавать игровые объекты в обход `GameObjectFactories`.

## 10. Мини-чеклист ревью новой фичи

1. Есть ли четкое разделение `System/Mechanic/Component/Container`?
2. Определены ли стейты активации системы?
3. Все Unity-ссылки находятся в контейнере?
4. Создание объектов идет только через `GameObjectFactories`?
5. Есть ли корректный `Dispose`?
6. Обновлена ли документация по архитектуре и стейтам?

## 11. Текущая реализация v1: GameplayTimer

Добавлен модуль `GameplayTimer` в `Assets/Scripts/GameSystemsScripts/GameplayTimer`:

- `GameplayTimerSystem`
- `GameplayTimerMechanic`
- `GameplayTimerComponent`
- `GameplayTimerContainer`

Назначение:

- считать игровое время текущей сессии;
- отображать его на HUD в формате `MM:SS`;
- не сохранять время между запусками в v1.

Правило активации:

- система обновляется только в `GameState.Gameplay`.

Таблица активации:

| System | Active State | Поведение |
| --- | --- | --- |
| `GameplayTimerSystem` | `Gameplay` | Инкрементирует время на `deltaTime` и обновляет HUD |
